# AGENTS.md

이 레포는 `프롬프트 모음`이 아니라 `문서 작성 스타일 킷`입니다. 따라서 새 프리셋이나 새 규칙을 추가할 때는 아래 원칙을 우선합니다.

## 0. 익명화 우선

git tracked 파일에는 실제 문서 제목, 제품명, 프로젝트명, URL, page id를 남기지 않습니다. 관련 기준은 [`PRIVACY.md`](./PRIVACY.md)를 따릅니다.

커밋 전에는 로컬 privacy check를 수행해 남아 있는 식별자가 없는지 확인합니다. 점검 도구는 로컬 전용 파일로만 운영하고 git tracked 영역에는 두지 않습니다.

## 1. 규칙 추가보다 회귀 확인 우선

새 규칙을 넣기 전에 기존 회귀 세트에서 어떤 실패가 반복되는지 먼저 확인합니다. 느낌상 더 좋아 보인다는 이유만으로 규칙을 늘리지 않습니다.

## 2. Skill Off / Skill On 비교 유지

가능하면 같은 문서를 기준으로 `Skill Off / Skill On` 비교를 남깁니다. 스킬 효과는 단일 결과물 감상보다 비교 결과로 판단합니다.

## 3. 실패 코드를 먼저 남김

문제가 보이면 자유서술부터 길게 쓰지 말고 [`evaluation/failure-taxonomy.md`](./evaluation/failure-taxonomy.md) 코드부터 붙입니다. 반복 실패가 2회 이상 나오면 그때 규칙 반영을 검토합니다.

특히 `F2`와 `F11`이 여러 문서에서 반복되면, 새 프리셋을 늘리기 전에 `첫 화면 압축`과 `표 기본값` 규칙을 먼저 강화합니다.

## 4. 문서 타입 프리셋을 함부로 늘리지 않음

새 프리셋은 실제 문서 타입에서 반복적으로 같은 구조 문제가 나올 때만 추가합니다. 기존 프리셋의 예시 보강만으로 해결되면 프리셋은 늘리지 않습니다.

## 5. README와 SKILL을 같이 관리

스킬 규칙을 바꿨으면 README의 설명, 문서 타입 표, evaluation 설명도 같이 맞춥니다. 공개 레포 사용자는 README를 먼저 보므로, README와 SKILL이 어긋나면 안 됩니다.

README에는 사용자 기준 설치와 사용 방법을 우선 적습니다. 배포 단위, publish 경로, 레지스트리 제출 같은 maintainer 관점 내용은 AGENTS 또는 로컬 운영 메모로 내립니다.

## 6. 실제 문서 기준 유지

예시는 추상적인 더미 문장보다 실제 업무 문서에서 자주 나오는 실패 패턴을 기준으로 고칩니다. 목표는 “좋은 글쓰기”가 아니라 “더 빨리 읽히는 보고용 문서”입니다.

## 7. OpenClaw / ClawHub 배포 메모

OpenClaw / ClawHub 관점에서는 repo 전체보다 `skills/report-memo-doc-style/` 폴더가 실제 배포 단위에 가깝습니다. 사용자 README에는 이 내부 운영 관점을 길게 쓰지 않고, maintainer가 배포할 때만 참고합니다.
